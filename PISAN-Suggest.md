## Project Summary

UWBSmartScheduler is a natural-language course-planning tool for UW Bothell CSS students. Students describe constraints in plain English ("only Tue/Thu, max 14 credits, include CSS 342") and a FastAPI backend parses the constraints, retrieves matching sections from a scraper-built course cache, and returns a conflict-free schedule rendered by a React frontend. Tech stack: Python/FastAPI + BeautifulSoup scraper + optional OpenAI embeddings/Pinecone vector store + React 18 SPA, deployed via Vercel (serverless function `api/index.py` + static React build). Contributors per `git shortlog`: Richie Chen (6 commits), Kevin Vo / khkvo (6 commits), Yousuf Al-Bassyiouni (1 commit, but it is the largest single contribution by line-count and includes the scraper rewrite, parser/conflict fixes, and frontend redesign).

## Suggested Improvements & New Features

### Code Quality & Technical
- Add the deployed Vercel URL to the very top of `README.md`. Right now there is no way for a grader (or another student) to find the live site. Also remove the orphan `# SmartScheduler` line at `README.md:260`.
- Consolidate documentation. `ARCHITECTURE.md` and `docs/ARCHITECTURE.md` are duplicated; `DELIVERABLES.md`, `SCRAPER_START_HERE.md`, `SCRAPER_IMPLEMENTATION_SUMMARY.md`, `FILES_OVERVIEW.txt`, and `TODO.md` are largely LLM-flavored marketing prose. Keep `README.md`, `ARCHITECTURE.md`, `SETUP.md`, `DEPLOY.md`, and delete the rest (or move them into `docs/archive/`). A reviewer who has to wade through 14 markdown files to find the truth will under-rate the project.
- Add a `.github/workflows/ci.yml` that at minimum runs `pytest backend/` and `cd frontend && npm test -- --watchAll=false`. The `backend/eval/run_eval.py` harness is the perfect fit for CI — running it in GitHub Actions would give you a green badge that proves the proposal's success criteria still pass.
- Remove the `node_modules/` history bloat. Even though it's no longer in `HEAD`, commit `1dd452b3` still contains it and every `git clone` pulls it down. Either `git filter-repo --path node_modules --invert-paths` (with team agreement) or just acknowledge it as tech debt.
- `backend/app/rag/conflict_checker.py:188` reads `meeting.get('days', [])` but `_pick_section` in `schedule_builder.py:151-158` reads `mt.get("days") or []`. Pick one defensive pattern and apply it consistently — the codebase mixes both.
- The CORS policy `allow_origins=["*"]` (`backend/main.py:29`) is fine for a class project but should be tightened to the Vercel domain before any "production" claim.
- `_extract_no_online` in `constraint_parser.py:285-308` has very loose regex (`(?:no|avoid|not|...)`) that fires on a lot of unrelated phrases — the query "I prefer morning classes" would match `(?:morning)` only because of the parser's other rules, but the no_online detection here is essentially a coin flip. Worth a unit test pass.
- `backend/app/rag/rag_pipeline.py:170-180` lists the prompt as `"Build a schedule..."` but the deterministic builder already did so — when the RAG message overrides, the LLM is being asked to do work that has already been done. Consider re-prompting it instead with "*explain* this already-built schedule in 1-2 sentences for the student", which is what the system actually wants.

### UI / UX Improvements
- The `apiStatus === 'error'` banner (`frontend/src/App.js:99-108`) tells a *student* to `cd backend && source venv/bin/activate`. That's a developer instruction in a student-facing UI. If the deployed backend is down, the user should see "We're having trouble — try again in a moment" instead.
- `CompletedCourses` is always rendered in the schedule view but a first-time visitor has no idea why. Add a one-line explainer ("Tick courses you've already passed so we don't recommend them again") or a tooltip.
- The hero copy in `App.js:115-124` is good but should include an explicit "Built for UW Bothell CSS students" tag *and* a one-line example of input/output. Right now a visitor has to read three sections before they understand what to type.
- Consider a screenshot or short demo gif at the top of `README.md`. For a visual product, a single image is worth more than the 250-line README.
- The example chips in `QueryInput.js:21-26` are great but all four are written in the same shorthand style. Add at least one *full-sentence* example so users see they can write naturally.

### New Features
- **Save/share schedules.** Right now nothing persists; once the page reloads, the schedule is gone. A localStorage save (and optionally a shareable URL with constraints encoded as a query string) is a half-day feature with huge UX payoff.
- **iCal/Google Calendar export.** Given that you already have section meeting times in `recommended_courses[].sections[].meeting_times`, generating an `.ics` file is straightforward and is exactly the kind of "concrete student utility" the rubric rewards.
- **MyPlan-style "what-if" mode.** Let the student tick courses they *plan* to complete next quarter, not just ones already done, and watch the recommendation graph open up.
- **Expand beyond CSS.** The scraper already supports all three campuses and 13+ departments per `DELIVERABLES.md`, but the `/api/schedule` endpoint and `CSS_CORE` set in `schedule_builder.py:26-29` lock it to CSS Bothell. Surfacing a department picker would directly improve the UW Community Impact score.
- **Waitlist / "is it open right now" indicator.** The scraper already pulls enrollment capacity and current enrollment (`DELIVERABLES.md:147-149`); surface that in the section card so students see "12/30 open" before they pick.

### AI Integration Enhancements
- **Move the LLM from message-only to constraint extraction.** The regex parser in `constraint_parser.py` is impressively detailed (~310 lines) but brittle — every new phrase is another regex. Replace the parser entry point with a *function-calling* LLM call that returns the same structured-constraints JSON. The deterministic builder downstream stays identical, but the system gains real natural-language understanding. This is the single biggest change that would push AI Integration from 11/15 toward 14/15.
- **Use the embeddings for course *discovery*, not just retrieval.** Right now `vector_store.query` returns top-k codes that are then filtered. Instead, let the student type "I like building things and hate writing essays" and use the embeddings to surface CSS electives that match the vibe — that is the kind of "AI-only-can-do-this" feature a chatbot wrapper cannot replicate.
- **Add an LLM-powered "explain why" link** on each recommended course card: a small button that calls a focused prompt with the course description + the student's stated constraints and renders a 2-sentence rationale. Cheap, transparent, and demonstrably AI.
- **Evaluation rigor.** `backend/eval/run_eval.py` and `eval_queries.json` (10 queries) are excellent — extend to 30-50 queries and add LLM-graded recall: "Did the LLM-only path produce the same conflict-free schedule as the deterministic builder?" That comparison is a publishable artifact and a strong piece for a peer-review/final-presentation slide.
